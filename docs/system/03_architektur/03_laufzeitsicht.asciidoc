[[sec:LS_benutzer-login]]
== [[sec:LS_benutzer-login]] Ablauf: Benutzer-Login (LF60/LF61)

Dieser Ablauf zeigt, wie sich ein Benutzer mit seinem Google-Konto im System anmeldet.

[plantuml, LS_benutzer_login, png]
----
@startuml
skinparam sequenceMessageAlign center
skinparam boxPadding 10

actor Benutzer

box "UI Layer" #LightBlue
    participant "LoginScreen" as UI
end box

box "Logic Layer" #E6F2FF
    participant "LoginSteuerung" as LoginCtrl
end box

box "Data Layer" #F5F5F5
    participant "AuthProvider" as Auth
    participant "BenutzerVerwaltung" as BenutzerMgmt
    participant "FirestoreClient" as Firestore
    participant "GoogleAuth" as GAuthWrapper
end box

box "External Services" #LightGray
    database "Firebase Auth" as FirebaseAuth
    database "Firestore DB" as FirestoreDB
end box

Benutzer -> UI: Klick "Mit Google anmelden"
activate UI

UI -> LoginCtrl: login()
activate LoginCtrl

LoginCtrl -> Auth: signInWithGoogle()
activate Auth

Auth -> GAuthWrapper: getGoogleIdToken()
activate GAuthWrapper
GAuthWrapper --> Auth: GoogleAuthCredential
deactivate GAuthWrapper

Auth -> FirebaseAuth: signInWithCredential(credential)
activate FirebaseAuth
alt Authentifizierung erfolgreich
    FirebaseAuth --> Auth: FirebaseUser
else Authentifizierung fehlgeschlagen
    FirebaseAuth --> Auth: Fehler
    Auth --> LoginCtrl: Fehler
    deactivate Auth
    deactivate LoginCtrl
    UI -> Benutzer: Fehlermeldung anzeigen
    deactivate UI
    return
end
deactivate FirebaseAuth

note right of Auth
Private Methode:
updateTokenOnServer(user)
end note

Auth -> Firestore: getDocument("users/{uid}")
activate Firestore
Firestore -> FirestoreDB: Read
FirestoreDB --> Firestore: Result
Firestore --> Auth: User? (Existierend oder null)
deactivate Firestore

alt User existiert nicht (null)
    note right of Auth: Fallback auf Rolle STUDENT
else User existiert
    note right of Auth: Bestehende Rolle übernehmen
end

Auth -> Firestore: setDocument("users/{uid}", user)
activate Firestore
Firestore -> FirestoreDB: Write / Merge
FirestoreDB --> Firestore: Success
Firestore --> Auth: Result<Unit>
deactivate Firestore

Auth --> LoginCtrl: Result<User>
deactivate Auth

LoginCtrl -> BenutzerMgmt: loadUser(userId)
activate BenutzerMgmt

BenutzerMgmt -> Firestore: getDocument("users/{uid}")
activate Firestore
Firestore -> FirestoreDB: Read
FirestoreDB --> Firestore: User Data
Firestore --> BenutzerMgmt: User Object
deactivate Firestore

BenutzerMgmt --> LoginCtrl: Result<User>
deactivate BenutzerMgmt

LoginCtrl --> UI: Result<User>
deactivate LoginCtrl

alt Login erfolgreich
    UI -> Benutzer: Navigation zum Dashboard
else Login fehlgeschlagen
    UI -> Benutzer: Fehlermeldung anzeigen
end

deactivate UI
@enduml
----

*Beschreibung:*
Der Anmeldevorgang wird durch den Benutzer in der UI initiiert. Die `LoginSteuerung` delegiert den Aufruf an den `AuthProvider`. Dieser führt zunächst die technische Authentifizierung über Firebase Authentication (Google OAuth2) durch.

Im selben Schritt stellt der `AuthProvider` sicher, dass ein korrespondierendes Benutzerprofil in der Firestore-Datenbank existiert:

* Existiert der Benutzer noch nicht, wird er automatisch mit der Rolle `STUDENT` angelegt.
* Existiert er bereits, werden die Basisdaten (z. B. Name, E-Mail) aktualisiert.

Anschließend lädt die `LoginSteuerung` über die `BenutzerVerwaltung` das Profil aus der Datenbank, um die aktuelle Rolle zu erhalten. Abschließend routet die UI je nach Ergebnis auf das passende Dashboard.

    note right of ProjektCtrl: 2. ID-Generierung (PRJ-Random)
    ProjektCtrl -> ProjektCtrl: generiereProjektId()

    ProjektCtrl -> ProjektRepo: save(project)
    activate ProjektRepo

    ProjektRepo -> FirestoreClient: setDocument("projects/{id}", project)
    activate FirestoreClient
    FirestoreClient -> CloudDB: SET Document
    activate CloudDB
    CloudDB --> FirestoreClient: Success
    deactivate CloudDB
    FirestoreClient --> ProjektRepo: Result.success
    deactivate FirestoreClient

    ProjektRepo --> ProjektCtrl: Result<Project>
    deactivate ProjektRepo

    ProjektCtrl --> UI: Result.success(project)
    deactivate ProjektCtrl

    ProjektCtrl --> UI: Result.failure
    UI -> Student: Zeige Fehlermeldung
end

@enduml
----

*Beschreibung:*
Der Student wählt in der UI einen lokalen Projektordner oder eine ZIP-Datei aus. Die UI übergibt die Binärdaten an die `ProjektSteuerung`, welche die Validierung übernimmt.

Anschließend orchestriert die `ProjektSteuerung` den Upload in zwei Schritten:

1. Binärdaten werden über `ProjektStorage` (→ `FirebaseStorageClient`) in Firebase Storage hochgeladen.
2. Erst nach erfolgreichem Upload werden die Metadaten über `ProjektVerwaltung` in Firestore gespeichert.

So werden „Dangling References“ vermieden.

[[sec:LS_batch-analyse]]
== [[sec:LS_batch-analyse]] Ablauf: Batch-Analyse (LF80)

[plantuml, LS_batch_analyse, png]
----
@startuml
skinparam sequenceMessageAlign center
skinparam boxPadding 10

actor Dozent

box "CodeScope Client (Intern)" #LightBlue
    participant "UI" as UI
    participant "AnalyseSteuerung" as AnalyseCtrl
end box

box "Logic / Data Layer" #E6F2FF
    participant "ProjektVerwaltung" as ProjektRepo
    participant "RestClient" as Rest
    participant "FirebaseFunctions" as FFClient
end box

box "Backend (Extern)" #LightGray
    participant "Cloud Function (analyseCode)" as CF
end box

Dozent -> UI: Wählt Projekte & Klick "Batch Start"
activate UI

UI -> AnalyseCtrl: startBatchAnalysis(projectIds, catalogId, model)
activate AnalyseCtrl

loop Für jede projectId in projectIds
    AnalyseCtrl -> ProjektRepo: findById(projectId)
    activate ProjektRepo
    ProjektRepo --> AnalyseCtrl: Project
    deactivate ProjektRepo

    alt Projekt bereit
        AnalyseCtrl -> Rest: requestAnalysis(projectId, catalogId, model)
        activate Rest
        Rest -> FFClient: httpsCallable("analyseCode")
        activate FFClient
        FFClient -> CF: Invoke
        activate CF
        CF --> FFClient: { analysisId }
        deactivate CF
        FFClient --> Rest: Response
        deactivate FFClient
        Rest --> AnalyseCtrl: Result.success(analysisId)
        deactivate Rest

        AnalyseCtrl -> ProjektRepo: save(project.copy(status=ANALYSIS_RUNNING))
        activate ProjektRepo
        ProjektRepo --> AnalyseCtrl: Success
        deactivate ProjektRepo
    else Fehler
        AnalyseCtrl --> UI: Result.failure("Batch failed for project X")
        break
    end
end

AnalyseCtrl --> UI: Result.success(List<analysisId>)
deactivate AnalyseCtrl

UI -> Dozent: Zeige Erfolgsmeldung
deactivate UI
@enduml
----

*Beschreibung:*
Der Dozent startet mehrere Analysen nacheinander. Die `AnalyseSteuerung` iteriert über Projekt-IDs, ruft pro Projekt die Cloud Function auf und setzt den Status. Bei einem Fehler kann der Batch abbrechen.

[[sec:LS_zusammenhang-ls]]
== [[sec:LS_zusammenhang-ls]] Zusammenfassung der Laufzeitsicht

Die dargestellten Szenarien verdeutlichen die Umsetzung einer Cloud-Native-Architektur, die stark auf entkoppelte Dienste und Firebase-Infrastruktur setzt. Wesentliche Muster:

* Zweistufige Authentifizierung: technischer Login (Firebase Auth) + fachliche Rollen/Profilverwaltung (Firestore).
* Direct-to-Cloud Upload: große Dateien werden direkt in Firebase Storage hochgeladen; Metadaten werden anschließend in Firestore gespeichert.
* Asynchroner Analyse-Start über Cloud Functions: Analyse wird serverseitig ausgeführt, Ergebnisse werden in Firestore persistiert.
* Batch-Verarbeitung: Mehrfachanalysen werden clientseitig iteriert angestoßen, um die Steuerung transparent zu halten.
* Berechtigungsprüfung vor Freigabe/Änderung: Dozentenoperationen werden gegen Projekt-/Owner-Informationen validiert.

Alle Komponenten kommunizieren über die in <<sec:komponenten, Kapitel 3.2>> definierten Schnittstellen und nutzen Firebase als zentralen Integrationspunkt.
